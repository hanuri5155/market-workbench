# PR 작업 흐름

이 저장소에서 사용하는 작업 흐름 정리용.

## 브랜치 규칙

- `feat/<topic>`: 기능 변경
- `fix/<topic>`: 버그 수정
- `docs/<topic>`: 문서 변경
- `chore/<topic>`: 기능 변화 없는 정리
- `ops/<topic>`: 배포, 운영, 워크플로우, 인프라 변경

브랜치 이름은 영어, 숫자, 하이픈만 사용.

## 기본 흐름

- `main`은 항상 배포 가능한 상태 유지
- `main` 반영은 PR 기준으로만 진행
- PR 1개에는 작업 목적 1개만 담기
- PR 머지 후 작업 브랜치 삭제
- 배포 태그는 `main` 기준으로만 생성

## 커밋과 PR

- 커밋 메시지 형식은 `type(scope): 한글 요약`
- `type(scope)`는 영어, 콜론 뒤 요약은 한글
- PR 제목은 변경 목적이 바로 드러나게 작성
- PR 본문은 `변경 요약`, `의도` 두 항목 중심으로 정리
- PR 생성 전 diff를 직접 확인하고 셀프 리뷰 수행

## 머지 방식

- `main` 반영은 squash merge로 통일
- 배포 영향이 큰 변경은 기능 변경과 분리
- 성격이 다른 변경이 섞이면 PR도 분리

## 분리 기준

- `docs/`: 문서만 포함
- `ops/`: 배포, 운영, 워크플로우, 인프라만 포함
- `feat/`: 기능 변경만 포함
- `fix/`: 버그 수정만 포함
- `chore/`: 기능 변화 없는 정리만 포함
